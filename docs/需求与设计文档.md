## 一、项目概述

### 1.1 背景

本项目为证券公司金融科技分公司内部羽毛球活动的比分记录与管理系统。团队经常组织羽毛球活动，需要一个便捷的工具来记录比赛结果、追踪球员积分和统计数据，替代手动记录或微信群内统计的方式。

### 1.2 目标

- 提供移动端友好的比赛计分体验（单打/双打）
- 建立公平的积分体系，激发参与热情
- 自动生成统计数据和排行榜
- 支持快速录入和语音式文字输入（连写名字自动拆分）
- 数据云端同步，支持多人使用

### 1.3 技术栈

| 技术 | 版本 | 用途 |
|------|------|------|
| React | 19.2.4 | UI 框架 |
| TypeScript | 5.x | 类型安全 |
| Vite | 8.0.3 | 构建工具 |
| Zustand | - | 状态管理（持久化到 localStorage） |
| Supabase | - | 云端数据库（PostgreSQL） |
| pinyin-pro | - | 拼音搜索与匹配 |
| recharts | - | 图表可视化 |
| Tailwind CSS | -（未使用） | — |
| CSS Variables | - | 主题系统（亮/暗模式） |

## 二、功能需求

### 2.1 比赛管理

#### 2.1.1 手工录入模式

- **比赛类型**：单打（1v1）/ 双打（2v2）
- **局数设置**：单局赛 / 三局两胜 / 五局三胜
- **计分方式**：
  - 逐球计分：点击 +1 按钮逐球记录，支持撤销
  - 直接输入比分：手动输入每局最终比分（如 21:13）
- **比赛日期**：可选择日期（默认今天），支持补录历史比赛
- **队伍选择**：球员网格选择，支持搜索（中文 + 拼音），已选球员自动排除
- **比赛保护**：有更早日期的未清算比赛时，阻止录入新比赛

#### 2.1.2 快速录入模式

- 输入格式：`张三李四 对 王五赵六 比分21:13`
- **智能名字拆分**：支持连写名字自动识别（如"王鹏万蕊清" → "王鹏" + "万蕊清"）
- **拼音匹配**：同音字模糊匹配（如输入"zhangsan"能匹配"张三"）
- **未识别球员处理**：弹出候选列表供确认，或一键新增球员
- 比分合法性校验（21分制规则）
- 自动计算积分变动

#### 2.1.3 计分板

- 显示当前局比分和总比分（X:Y）
- 各局比分条（当前局高亮，已结束局显示胜负标记）
- 比赛结束后显示获胜者动画和积分变动详情
- 支持撤销（逐球模式，基于快照恢复）
- 支持恢复上次未完成的比赛

### 2.2 球员管理

- 球员 CRUD（增删改查）
- 搜索支持中文和拼音
- 显示每个球员的：级别标签、比赛场次、胜率、积分
- **图形化统计**（点击球员展开）：
  - 总场次/胜场/负场概览卡片
  - 胜负比例环形图
  - 胜负趋势面积图
  - 每场比赛得分对比柱状图
  - 对手战绩堆叠柱状图
  - 活跃时段分布图

### 2.3 积分排行榜

- 按积分从高到低排序
- 显示：排名、姓名、级别（L0~L4）、比赛场次、胜负、积分
- 前三名金银铜视觉区分
- 级别按积分排名动态划分

### 2.4 比赛历史

- 按时间倒序展示已完成的比赛
- 点击查看详情（各局比分、积分变动）
- 单条删除（自动回退积分）
- **批量多选删除**（全选/反选，批量删除并回退积分）

### 2.5 统计分析

- **概览**：总比赛场次、总局数、活跃玩家数、单打场次
- **胜率排行**：所有有比赛记录的玩家（含已删除），胜率进度条
- **对阵记录查询**：选择两个玩家查看 H2H 战绩
- **活动频率**：近30天热力图（GitHub 贡献图风格）

### 2.6 比赛日清算系统

- **日终清算**：选择日期，系统基于上一个已清算快照重新计算当日积分，与实时数据对比
- **清算确认**：确认后保存快照（DaySnapshot），标记该日已清算
- **回滚**：删除指定日期的所有比赛，恢复球员积分到该日期之前的状态
- **未清算提醒**：首页自动检测未清算日期并提示

### 2.7 设置

- **主题切换**：亮色 / 暗色模式（CSS Variables）
- **积分重置**：所有球员积分恢复初始值（2000），需二次确认
- **数据导出**：JSON 格式（含版本号、球员、比赛）
- **数据导入**：支持版本校验、数据合法性校验（10MB/10000条上限）
- **清除所有数据**：不可撤销操作

## 三、数据模型设计

### 3.1 核心类型定义

```typescript
// 球员
interface Player {
  id: string;          // UUID
  name: string;
  avatar?: string;
  createdAt: number;   // 时间戳
  rating?: number;     // 积分，初始 2000
}

// 比赛
interface Match {
  id: string;
  type: 'singles' | 'doubles';
  mode: 'single' | 'best-of-3' | 'best-of-5';
  scoreMode: 'point-by-point' | 'direct-input';
  team1: Team;
  team2: Team;
  currentGame: number;
  games: GameScore[];
  status: 'ongoing' | 'completed';
  winner?: 'team1' | 'team2';
  createdAt: number;
  completedAt?: number;
  scoreHistory: ScoreHistoryEntry[];
  ratingChanges?: RatingChange[];
  matchDate?: string;  // "2026-03-27"
}

interface Team {
  players: Player[];
  score: number;
  gamesWon: number;
}

interface GameScore {
  team1Score: number;
  team2Score: number;
  winner?: 'team1' | 'team2';
}

interface RatingChange {
  playerId: string;
  playerName: string;
  delta: number;         // 正数加分，负数减分
  ratingBefore: number;
  ratingAfter: number;
}

// 日清算快照
interface DaySnapshot {
  id: string;
  date: string;           // "2026-03-27"
  createdAt: number;
  playerSnapshots: PlayerSnapshot[];
  matchIds: string[];
  status: 'settled' | 'pending';
}

interface PlayerSnapshot {
  playerId: string;
  name: string;
  rating: number;
  level: number;
  totalWins: number;
  totalLosses: number;
  totalMatches: number;
}
```

### 3.2 状态管理（Zustand Store）

全局状态存储在 `store/index.ts`，使用 `zustand/persist` 持久化到 `localStorage`（key: `badminton-store`）。

**存储内容**（partialize）：
- `players`：球员列表
- `matches`：比赛列表（scoreHistory 被清除以节省空间）
- `theme`：主题设置
- `daySnapshots`：清算快照列表
- `currentMatch`：始终为 null（不持久化进行中的比赛）

**Store Actions**：

| 分类 | 方法 |
|------|------|
| 通知 | `showNotification`, `clearNotification` |
| 数据加载 | `loadFromCloud` |
| 球员 | `addPlayer`, `updatePlayer`, `deletePlayer` |
| 比赛 | `createMatch`, `addScore`, `setGameScore`, `undoScore`, `finishMatch`, `deleteMatch`, `deleteMatches`, `loadMatch` |
| 积分 | `resetAllRatings` |
| 清算 | `getUnsettledDates`, `settleDay`, `confirmSettlement`, `rollbackDay` |
| 其他 | `toggleTheme`, `importData`, `clearAllData` |

### 4.3 数据流

```

                ↓
         Supabase（云端持久化）
                ↓
         localStorage（本地备份）
```

**云优先策略**：
1. 应用启动时 `loadFromCloud()` 从 Supabase 拉取数据
2. 所有写操作先更新本地状态（乐观更新），再同步到云端
3. 云端失败时回滚本地状态并显示错误通知

## 六、核心算法

### 6.1 羽毛球计分规则

```typescript
// 21分制
WINNING_SCORE = 21
MAX_SCORE = 30
MIN_WIN_DIFF = 2

// 判断单局胜负
isGameWon(score1, score2):
  - 任一方 ≥ 21 且分差 ≥ 2 → 胜
  - 任一方 = 30 → 胜（30:29 决胜）

// 判断整场胜负
isMatchWon(gamesWon, mode):
  - single: 直接判定
  - best-of-3: gamesWon ≥ 2
  - best-of-5: gamesWon ≥ 3
```

### 6.2 比分校验

`validateGameScore(score1, score2)` 完整规则：

| 条件 | 结果 |
|------|------|
| 负数 | ❌ 不合法 |
| 平局 | ❌ 不合法 |
| > 30 | ❌ 不合法 |
| 胜方 = 30 | ✅ 仅允许 30:29 |
| 胜方 = 21 | ✅ 败方 0~19，分差 ≥ 2 |
| 胜方 22~29 | ✅ 败方必须 ≥ 20，分差 ≥ 2 |

### 6.3 积分系统

**初始积分**：2000 分

**级别划分**（基于积分排名，≥ 5人时生效）：

| 级别 | 区间 | 说明 |
|------|------|------|
| L0 | 前 20% | 顶级选手 |
| L1 | 20%~40% | 优秀 |
| L2 | 40%~60% | 中等（不足5人时全部归此级） |
| L3 | 60%~80% | 初级 |
| L4 | 后 20% | 新手 |

**积分变动规则**（零和博弈）：

| 级别差 | 弱方赢 | 强方赢 |
|--------|--------|--------|
| 0 | +50 / -50 | +50 / -50 |
| 1 | +65 / -65 | +35 / -35 |
| 2 | +75 / -75 | +25 / -25 |
| 3 | +85 / -85 | +15 / -15 |
| 4 | +90 / -90 | +10 / -10 |

**双打处理**：取队伍平均级别进行比较。

### 6.4 快速录入 — 名字拆分算法

```typescript
// splitNames("王鹏万蕊清", players) → ["王鹏", "万蕊清"]

function splitNames(input, players):
  backtrack(start):
    尝试 len = 2~5 的子串
    精确匹配球员名 或 拼音匹配（长度差 ≤ 1）
    递归尝试剩余部分
  return 成功拆分 或 [原始输入]
```

### 6.5 日清算算法

```
1. 找到上一个已清算快照（date < targetDate）
2. 以快照中的球员数据为基准
3. 按时间顺序重放当日所有比赛
4. 计算每个球员的清算后积分
5. 与实时积分对比 → consistent / inconsistent
6. 确认后保存快照
```

## 七、存储方案

### 7.1 双层存储架构

| 层 | 技术 | 用途 |
|----|------|------|
| 云端 | Supabase (PostgreSQL) | 主存储，多设备同步 |
| 本地 | localStorage | 离线备份，快速加载 |

### 7.2 Supabase 数据表

| 表 | 主要字段 | 说明 |
|----|----------|------|
| `players` | id, name, avatar, created_at, rating | 球员信息 |
| `matches` | id, type, mode, team1(team JSON), team2, games, status, winner, rating_changes, match_date | 比赛记录 |
| `day_snapshots` | id, date, player_snapshots(JSON), match_ids, status | 清算快照 |

### 7.3 乐观更新策略

```
操作 → 更新本地状态 → 异步写入云端
                           ↓ 成功
                       （无操作）
                           ↓ 失败
                       回滚本地状态 + Toast 错误
```

## 九、已知限制与后续规划

### 9.1 已知限制

1. **多用户隔离**：当前 Supabase 未启用 RLS（行级安全），所有用户共享数据
2. **离线模式**：localStorage 仅作备份，离线时写入操作会失败
3. **scoreHistory 持久化**：持久化时清除了 scoreHistory（节省空间），导致撤销功能在刷新后失效
4. **PWA 未完成**：Service Worker 和离线缓存未完全配置
5. **统计页面**：从导航中移除（当前 5 个 Tab 不含统计），但组件代码保留
6. **Tailwind CSS**：项目未实际使用 Tailwind，纯 CSS Variables + 自定义样式

### 9.2 后续规划建议

- [ ] 启用 Supabase RLS 实现多用户数据隔离
